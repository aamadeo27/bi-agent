# Functional Requirements Specification: BI Result Presenter Webapp

## Access Control & Security
* The system should support Role-Based Access Control (RBAC) with customizable user-defined roles.
* The system should provide an administration interface for administrators to create, modify, and delete roles.
* The administrator should be able to assign data access permissions to roles at the schema, table, row, or column level.
* The system should restrict the LLM agent's data retrieval capabilities to only the data sources and subsets permitted by the active user's assigned role.
* The system should evaluate permissions prior to executing any generated SQL queries or REST API requests to ensure unauthorized data is never exposed.

## Data Visualization & Charting
* The system should automatically determine the most appropriate chart type based on the structure and dimensions of the retrieved data.
* The system should support the following primary chart types for immediate implementation:
    * **Bar Charts (Periodic & Categorical):** For comparing discrete quantities or tracking metrics over time.
    * **Line Charts:** For displaying continuous data trends over time.
    * **Pie/Donut Charts:** For illustrating proportions and percentage breakdowns of a whole (limited to low-cardinality data).
    * **Data Tables:** For presenting raw tabular data, sorting, and pagination when graphical charts are not applicable.
* The customer should be able to toggle between a graphical chart view and a raw data table view for any generated result.
* The customer should be able to export generated charts as image files (PNG/JPEG) or data payloads (CSV/JSON).
* *Future Work:* The system should support advanced visualizations in future iterations, including **Scatter Plots** (for correlation analysis), **Stacked Area Charts** (for composition over time), and **Geospatial Heatmaps** (for regional data analysis).
* *Future Work:* The customer should be able to manually manipulate rendered charts using interactive UI controls (e.g., dropdowns to switch chart types manually or change color palettes).

## LLM Agent & BI Integration
* The system should translate natural language user queries into structured SQL queries or REST API requests depending on the target data source.
* The LLM agent should validate user queries against the BI system schema and permission constraints before execution to prevent syntax or security violations.
* The system should provide an explicit error state or ask for clarification if a user query is ambiguous, missing parameters, or references data outside the available schema.
* The system should stream the LLM's text response to the user in real time.
* The system should display the underlying data query or logic generated by the LLM (e.g., the executed SQL or API endpoint) upon user request for auditing purposes.

## User Interface & Interaction
* The customer should be able to input questions via a persistent conversational chat interface.
* The system should maintain conversation history context to allow for follow-up questions (e.g., "Filter the previous result by region X").
* The system should render interactive charts directly within the chat timeline as inline responses.
* The customer should be able to hover over chart elements (bars, lines, slices) to view precise tooltips with data point values.

---

### Note
When implementing this architecture, standard practice requires passing the schema metadata (table definitions or API documentation) to the LLM context. To prevent SQL injection or broad data exposure from unexpected LLM outputs, the database user or API token utilized by the webapp must be restricted at the infrastructure level to strictly match the permissions of the logged-in user.
