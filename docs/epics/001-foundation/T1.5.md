# CI pipeline (lint, typecheck, unit, integration, security-tests, build)

description: Create the full CI pipeline (GitHub Actions) that runs on every PR
and on merge to `main`. Stages: install (pnpm frozen lockfile) → lint → typecheck
→ unit tests (Vitest incl. adversarial security suites) → integration tests
(Supertest + Testcontainers Postgres incl. tenant-isolation test from T1.2) →
build (multi-stage Docker images for `apps/api` and static build for `apps/web`)
→ security-tests (adversarial Vitest suites for permission gate, query validator,
tenant middleware, restricted-credential proxy). Wire Turborepo remote cache
(TURBO_TOKEN + TURBO_TEAM). PRs are green only if all stages pass.

tag: infra
deps: T1.1, T1.2, T1.3, T1.4

acceptance criteria:
- `.github/workflows/ci.yml` exists and runs the following jobs in order:
    install → lint → typecheck → unit → integration → build → security-tests.
- Each job reports a distinct status check on the PR (not a single monolithic check).
- `install` uses `pnpm install --frozen-lockfile`; fails if lockfile is out of sync.
- `lint` runs `turbo lint`; fails on any ESLint or Prettier violation, including
  the custom rule that forbids LLM provider SDK imports outside
  `apps/api/src/llm/adapters/*`.
- `typecheck` runs `turbo typecheck` (`tsc --noEmit`, `strict: true`).
- `unit` runs `turbo test:unit`; Vitest; all unit tests pass; coverage threshold
  enforced for security-critical modules (permission gate, query validator,
  tenant middleware, restricted-credential proxy).
- `integration` spins up a Testcontainers Postgres instance, runs
  `prisma migrate deploy` against it, and executes Supertest integration tests
  including the tenant-isolation test from T1.2 (cross-tenant id injection is
  rejected).
- `build` runs `turbo build`; both Docker images build successfully
  (`apps/api/Dockerfile` multi-stage, `apps/web/Dockerfile` Vite build +
  static export).
- `security-tests` job runs `turbo test:security` — the adversarial Vitest suites
  for the four security-critical modules (permission gate, query validator, tenant
  middleware, restricted-credential proxy). Any failure blocks merge.
- Turborepo remote cache is wired: `TURBO_TOKEN` and `TURBO_TEAM` are read from
  GitHub Actions secrets; cache hits are logged in CI output.
- Branch protection on `main` requires all six check names to be green before
  merge is allowed (document the exact check names in the repo README).
- `.env.example` committed with placeholder values; `.gitignore` blocks `.env`,
  `.env.local`, `*.pem`, `*.key` (CI lint step fails if these patterns appear in
  the diff).
- README documents `cp .env.example .env.local` and `pnpm dev` local startup.

kb-refs:
  conventions: [testing, naming, branching, secrets]
  tech-stack:  [typescript, postgres, docker]
  devops:      [ci-pipeline, environments, secrets]

notes:
- E2E (Playwright + axe-core) runs post-merge on staging, not on PR — add a
  separate `e2e` job that is triggered only on push to `main` (not required for
  PR merge gate). See devops.md §2.4.
- The `security-tests` job is a separate job (not merged into `unit`) so CI
  surfaces security failures with a distinct label.
- Turborepo task graph: `test:unit`, `test:integration`, `test:security`, `lint`,
  `typecheck`, `build` must all be defined in the root `turbo.json` with correct
  `dependsOn` entries (e.g. `build` depends on `typecheck`).
