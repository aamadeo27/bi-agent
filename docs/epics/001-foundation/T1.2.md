# Control-plane DB & schema-per-tenant migrations

description: Set up the **control-plane** PostgreSQL via Prisma + Prisma Migrate
(Prisma is control-plane only — never used for tenant data sources). Create the
shared `platform` schema (tenant registry, global config) and a **tenant-provisioning
service** that creates a `tenant_<id>` schema with the per-tenant tables (users,
roles, resource_grants, conversations, messages, data_sources, audit_events,
cred_vault refs). Use a **single PrismaClient** (one pool). Provide a request-time
`withTenant(tenantId)` helper that runs work inside a Prisma interactive transaction
and pins the connection `search_path` via `SET LOCAL search_path TO "tenant_<id>",
platform` (transaction-scoped so it cannot leak to the next pooled borrower). Wire
Testcontainers/disposable test schema for DB tests.

tag: backend
deps: T1.1

acceptance criteria:
- Prisma Migrate creates `platform` schema + provisioning routine for `tenant_<id>`.
- `provisionTenant(id)` creates a fully-formed tenant schema; idempotent.
- `withTenant(tenantId)` scopes work via `SET LOCAL search_path` inside a Prisma
  interactive transaction; the path does NOT persist on the pooled connection after
  the transaction (assert no leakage to a subsequent borrow).
- A single PrismaClient/one pool is used (no per-tenant client).
- Integration test: objects created in `tenant_A` are not visible from a
  `tenant_B`-scoped connection (FR-AC-6, NFR-SEC-4, GAP-6).
