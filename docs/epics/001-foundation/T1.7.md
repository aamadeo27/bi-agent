# docker-compose for local development (API + Postgres + seed)

description: Write `docker-compose.yml` at the repo root that brings up the full
local stack with a single `docker compose up`: a Postgres 16 service (control-plane
DB), the API service (built from `apps/api/Dockerfile`), and a one-shot seed/migrate
service that runs `prisma migrate deploy` against the `platform` schema and provisions
a demo tenant schema with a seeded admin role. The web SPA runs via `pnpm dev` (Vite
dev server) outside compose for fast HMR, but may optionally be added as a profile.
Local secrets come from `.env.local` (gitignored), never committed.

tag: infra
deps: T1.1, T1.2, T1.6

acceptance criteria:
- `docker-compose.yml` exists at repo root with services: `db` (postgres:16-alpine),
  `api` (build context `apps/api`, depends_on `db` healthy), and `migrate-seed`
  (one-shot: `prisma migrate deploy` + tenant/admin-role seed, exits 0).
- `db` service uses a named volume for persistence and exposes 5432 on localhost only;
  healthcheck uses `pg_isready`.
- `api` reads config from `.env.local` via `env_file`; no secrets are hardcoded in the
  compose file or any image layer.
- `migrate-seed` runs after `db` is healthy, applies control-plane migrations to the
  `platform` schema, and provisions one demo tenant schema (`tenant_demo`) with a
  seeded admin role that has the `can_inspect_query` capability ON.
- `docker compose up` brings the stack to a working state where the API responds on
  its health endpoint; `docker compose down -v` tears it down cleanly.
- `.env.example` documents every variable the compose stack needs (DATABASE_URL,
  GEMINI_API_KEY placeholder, JWT_ACCESS_SECRET, JWT_REFRESH_SECRET, KMS_KEY_REF).
- README documents the local-dev flow: `cp .env.example .env.local`,
  `docker compose up`, then `pnpm --filter web dev` for the SPA.

notes:
- Reuse the API image from T1.6 (`apps/api/Dockerfile`) — do not maintain a second
  Dockerfile for local.
- The `migrate-seed` provisioning step should call the same tenant-provisioning
  routine that T8.1 (`provision-tenant.ts`) exposes, so local and prod use one
  code path. If T8.1 is not yet merged, a thin inline seed is acceptable but must
  be replaced once the script lands.
- Keep the web SPA out of the default compose path (HMR is faster via host `pnpm dev`);
  an optional `--profile full` may add it later.
