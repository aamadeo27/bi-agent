# Dockerfiles — API (multi-stage) and Web (Vite build)

description: Write `apps/api/Dockerfile` (multi-stage Node 20 alpine: builder stage
runs `pnpm build`; runtime stage copies compiled output and production deps only,
runs as non-root user) and `apps/web/Dockerfile` (Vite build stage produces `dist/`;
serve stage uses `nginx:alpine` to serve static files). Both images must build
successfully in CI (`docker build --no-cache`) and produce minimal, non-root
runtime images. Verify image sizes are reasonable (API < 300 MB, Web < 50 MB).

tag: infra
deps: T1.1

acceptance criteria:
- `apps/api/Dockerfile` exists; multi-stage: stage `builder` uses `node:20-alpine`,
  installs pnpm, copies workspace source, runs `pnpm install --frozen-lockfile` and
  `pnpm build`; stage `runtime` copies compiled `dist/` and production-only
  `node_modules` from builder, runs `node dist/index.js` as a non-root user
  (`USER node` or equivalent).
- `apps/web/Dockerfile` exists; stage `builder` uses `node:20-alpine`, runs
  `pnpm install --frozen-lockfile` and `vite build`; stage `serve` uses
  `nginx:alpine`, copies `dist/` to `/usr/share/nginx/html`, serves on port 80.
- `docker build -t bi-api apps/api` and `docker build -t bi-web apps/web` both
  succeed with exit code 0 (run in CI `build` job).
- No secrets, `.env` files, or private key material is COPY-ed into any image layer;
  CI lint step (`docker history` scan or `.dockerignore` presence) enforces this.
- `.dockerignore` files exist at `apps/api/.dockerignore` and `apps/web/.dockerignore`
  excluding `node_modules`, `.env*`, `*.pem`, `*.key`, and test fixtures.
- Images run as non-root in both final stages.
- `docker images` shows API image < 300 MB and Web image < 50 MB (document sizes
  in PR description; not a hard CI gate but a soft target).
- `apps/api/Dockerfile` and `apps/web/Dockerfile` are referenced in the CI `build`
  job (T1.5) with `docker build` steps.

kb-refs:
  conventions: [naming, secrets]
  tech-stack:  [typescript, express, react, docker]

notes:
- The API Dockerfile must work for both local `docker build` and the CI `build` job;
  TURBO_TOKEN / TURBO_TEAM are NOT baked into the image — they are only available
  as CI job env vars at build time and must not appear in any COPY or ENV layer.
- For the web serve stage, include a minimal `nginx.conf` that rewrites all routes to
  `index.html` (required for React Router / client-side routing).
- Tag images as `:<git-sha>` in CI; `latest` tag is acceptable for local builds.
