# Tenant provisioning script (scripts/provision-tenant.ts)

description: Write `scripts/provision-tenant.ts`, the single code path that onboards a
tenant. Given a tenant id/slug it: registers the tenant in the `platform` tenant
registry, creates the `tenant_<id>` Postgres schema, applies the control-plane
migrations to that schema (`prisma migrate deploy` scoped to the new schema), and
seeds a tenant admin role with the `can_inspect_query` capability ON plus an initial
admin user invite. Idempotent — re-running for an existing tenant is a no-op (or a
safe repair), never a destructive reset.

tag: infra
deps: T1.1, T1.2 (control-plane schema + tenant registry)

acceptance criteria:
- `scripts/provision-tenant.ts` accepts a tenant id/slug (and optional admin email)
  via argv or a typed function export, usable from CLI and from other code (local
  compose seed in T1.7 calls the exported function).
- Registers the tenant row in the `platform` schema tenant registry.
- Creates schema `tenant_<id>` if absent; applies all control-plane migrations to it
  using `SET LOCAL search_path` / `prisma migrate deploy` against that schema (no
  bare `SET search_path` that could leak across pooled connections).
- Seeds exactly one admin role for the tenant with `can_inspect_query` = true and
  full schema/table/column grant scope; creates an admin user invite (status pending)
  for the provided email if given.
- Idempotent: a second run for the same tenant does not duplicate rows, drop data, or
  error; it reports "already provisioned" and optionally repairs missing seed rows.
- Wrapped in a transaction per tenant so a mid-run failure leaves no half-provisioned
  schema; logs each step.
- Unit/integration test: provisions a tenant against a Testcontainers Postgres,
  asserts schema + admin role + capability flag, then asserts a second run is a no-op.

notes:
- This is the single tenant-creation code path: T1.7 (compose seed) and any future
  super-admin/onboarding flow must call this, not re-implement provisioning.
- Row-level permissions are out of scope (v1) — seed grants at schema/table/column
  granularity only.
- Does NOT touch tenant data sources (those are registered later via epic 004); this
  script is control-plane only.
