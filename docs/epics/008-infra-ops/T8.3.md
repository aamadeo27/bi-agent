# Per-tenant migration authoring guide + CI lint check

description: Document how to author database migrations safely under the
schema-per-tenant model, and add a CI lint check that catches migrations which would
break tenant isolation. Because every tenant has its own `tenant_<id>` schema and
control-plane migrations are applied per schema, a careless migration (hardcoded
schema name, cross-schema reference, bare `SET search_path`, non-idempotent DDL) can
leak data across tenants or break provisioning. This task produces the guardrails.

tag: infra
deps: T8.1

acceptance criteria:
- `docs/kb/conventions/migrations.md` (or a clearly linked guide) documents: how
  schema-per-tenant migrations are applied (`prisma migrate deploy` per schema with
  `SET LOCAL search_path`), what is forbidden (hardcoded `tenant_*` names, references
  to another tenant's schema, bare `SET search_path`, destructive non-reversible DDL
  without a backfill plan), and the review checklist for any migration PR.
- The KB conventions index (`docs/kb/conventions/README.md`) links the new guide.
- A CI lint check (script run in the `lint` job) scans migration files and fails on:
  a literal `tenant_` schema name in a migration, a bare `SET search_path` (must be
  `SET LOCAL`), and cross-schema qualified references.
- The lint check has unit tests: a known-bad migration sample fails, a known-good one
  passes.
- Guide includes a worked example of adding a column to a per-tenant table that
  applies correctly across all tenant schemas.

kb-refs:

  tech-stack:  [postgres, prisma]

notes:
- Ties directly to the Prisma/credential reconciliation: Prisma is control-plane only
  and isolation depends on `SET LOCAL search_path` — this guide + lint enforce that at
  the migration layer.
- Keep the lint check fast (static scan of migration SQL/TS); it runs in the existing
  `lint` job, not a new CI job.
