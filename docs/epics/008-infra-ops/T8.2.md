# IaC stub — Cloud Run + Cloud SQL + Secret Manager + KMS

description: Author an infrastructure-as-code **scaffold** (Terraform recommended;
Pulumi acceptable) describing the runtime environment from devops.md: a Cloud Run
service for the API, Cloud SQL (Postgres 16) for the control plane, Secret Manager for
runtime secrets, and a KMS key for the credential-vault envelope encryption. Scope is
a stub teams extend — enough to `plan`/`validate` cleanly and document the resource
shape, not a production-hardened estate.

tag: infra
deps: 001

acceptance criteria:
- IaC lives under `infra/` with a clear module layout (e.g. `infra/main.tf`,
  `variables.tf`, `outputs.tf`, or Pulumi equivalent).
- Declares: Cloud Run service (API container, env from Secret Manager), Cloud SQL
  Postgres 16 instance, Secret Manager secrets (DATABASE_URL, GEMINI_API_KEY,
  JWT_ACCESS_SECRET, JWT_REFRESH_SECRET, KMS_KEY_REF), and a KMS key/keyring for the
  credential vault.
- No secret values committed — only secret *references*/placeholders; real values are
  injected out-of-band. `terraform validate` (or `pulumi preview` against a stub
  backend) passes in CI.
- Variables parameterize project id, region, and environment (dev/staging/prod) so the
  same module deploys all three.
- `infra/README.md` documents: what is stubbed vs production-ready, the assumed manual
  steps, and the deferred hardening (VPC connector, IAM least-privilege, automated
  backups, alerting).
- A `.gitignore` entry excludes local state and `*.tfvars` with real values.

notes:
- Intentionally a stub — do NOT over-build a full production estate. Hardening is
  deferred and called out in the README and the epic DESCRIPTION risks.
- Deploy target is a recommendation (Cloud Run + Cloud SQL); keep modules swappable so
  AWS ECS / Fly.io is a later substitution without app-code change.
- KMS key ref produced here is consumed by the credential vault in epic 004.
