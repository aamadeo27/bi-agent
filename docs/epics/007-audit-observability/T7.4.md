# Alert rule as code — P1 permission-gate bypass/error (A1)

description: Provision the **single security-critical alert A1** from `monitoring.md §3`
as code (config in the repo, not click-ops), bound to the wired OTel/metrics backend.
Scope is intentionally narrowed for v1 (user scope decision): only **A1 permission-gate
bypass/error (P1, never suppressed)** is implemented as code in v1. The remaining
alerts (A2 cross-tenant, A3–A12 SLO/failure/cost/streaming) and the dashboard (T7.5)
are deferred post-v1 and live in `monitoring.md` as a runbook to wire manually if
needed. Depends on T7.3 emitting the metrics.

tag: infra
deps: T7.3

acceptance criteria:
- A1 exists as committed config with the documented condition + threshold + window +
  severity: fires on any `gate.bypass.count` or `gate.error.count` > 0, routed P1.
- A test/dry-run asserts A1 triggers from a synthetic bypass metric and that A1 is NOT
  silenced by any deploy/maintenance suppression rule.
- The P1 notification channel placeholder (sec-alerts) is configured.
- Backend target is config-driven (matches T7.3 exporter) so swapping backends does not
  require rewriting rule logic.

- A3–A12 and the 6-panel dashboard (T7.5) are deferred post-v1; monitoring.md serves as
  the manual runbook for them.
