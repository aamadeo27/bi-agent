# Conversation retention purge job (365-day default)

description: Implement the scheduled retention purge that enforces GAP-4: hard-delete
any conversation (and its messages) whose age exceeds the **365-day** default
retention, per tenant. Provide a `purgeExpiredConversations({ olderThanDays })`
service (default 365, configurable via env) that runs across all tenant schemas, and
a scheduler entry (e.g. node-cron or an external cron hitting an internal admin
trigger) that invokes it on a daily cadence. Purge runs inside the correct tenant
`search_path` per tenant (Prisma `SET LOCAL` transaction pattern) and emits an audit
event summarizing counts per tenant (no row data). This complements — does not
replace — manual user delete (`DELETE /api/conversations/:id`).

tag: backend
deps: T5.6

acceptance criteria:
- `purgeExpiredConversations({ olderThanDays = 365 })` deletes conversations + their
  messages older than the cutoff, per tenant, idempotently; younger ones are kept.
- The retention window is read from config (default 365 days) — not hard-coded at
  the call site.
- Each tenant is processed inside its own `search_path` (no cross-tenant deletion);
  a per-tenant summary audit event is emitted (counts only, no row/PII data).
- A scheduler triggers the job daily; it is safe to run concurrently/again (no
  double-delete errors).
- Tests: deletes only expired rows, respects the configurable window, is tenant-
  scoped, is idempotent, and emits the summary audit event.
